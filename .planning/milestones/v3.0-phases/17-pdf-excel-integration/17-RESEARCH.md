## Summary

Phase 17 extends two existing services ‚Äî `services/pdf_report.py` (ReportLab Platypus) and `services/excel_report.py` (XlsxWriter) ‚Äî to include layout recommendation output generated by `pipeline/layout_engine.py`. Both services are already working and tested; this phase adds new content blocks without rewriting anything.

The PDF change adds a third page (or continuation of page 2) with a compact 4-column comparison table (Strategy | Consolidation | Performance | Uniform) covering the 15 `LayoutMetrics` fields visible in the UI's `_build_comparison_table()`. It appears after the existing charts section, separated by a `PageBreak()`. The header callback already runs on all pages, so no change is needed there.

The Excel change adds a fourth sheet named via a new `excel.sheet_layout` i18n key. The sheet contains a compact strategy-comparison summary table at the top, then three per-strategy sub-tables (one per `LayoutProposal.datastores`), each listing datastore name, raw capacity, used capacity, utilization%, VM count, IOPS, and workload types. The brand formats must be recreated locally since XlsxWriter `Format` objects are workbook-scoped and the new sheet helper must receive the workbook reference.

Both services need `generate_all_proposals()` called with the same `CalculationSummary` that feeds the rest of the report. The `PlacementConstraints` defaults are adequate for a report context (no session-state tuning needed ‚Äî the report always uses defaults). Testing follows existing patterns: PDF tests check `result[:5] == b"%PDF-"` and locale byte-size differences; Excel tests check `result[:4] == b"PK\x03\x04"` and sheet count via `zipfile`.

**Primary recommendation:** Extend `generate_report_pdf()` and `generate_report_xlsx()` in-place by adding layout-aware helper functions (`_build_layout_page()` and `_write_layout_sheet()`). Call `generate_all_proposals()` internally in each service ‚Äî do not change the public API signatures.

### Pattern 1: Adding a PDF Page via Platypus Story

The existing `generate_report_pdf()` builds a `story: list[Flowable]` then calls `doc.build(story, ...)`. A new page is added by appending to the story after the charts section:

```python
# Source: existing pdf_report.py lines 384-405 (charts section pattern)
# After the before/after bar chart:
story.append(PageBreak())
story.append(Paragraph(t("pdf.layout_heading"), heading_style))
story.append(Spacer(1, 12))

# Comparison table: header row + 1 row per metric
header = [t("layout_page.metric"), t("strategy.consolidation"), t("strategy.performance"), t("strategy.uniform")]
table_data = [header]
for metric_key, c_val, p_val, u_val in _layout_metric_rows(proposals):
    table_data.append([t(f"metrics.{metric_key}"), c_val, p_val, u_val])

col_widths = [160, 100, 100, 100]
table = Table(table_data, colWidths=col_widths)
table.setStyle(TableStyle([
    ("BACKGROUND", (0, 0), (-1, 0), _BRAND_BLUE),
    ("TEXTCOLOR", (0, 0), (-1, 0), colors.white),
    ("FONTNAME", (0, 0), (-1, 0), "VeraBd"),
    ("FONTNAME", (0, 1), (-1, -1), "Vera"),
    ("FONTSIZE", (0, 0), (-1, -1), 9),
    ("GRID", (0, 0), (-1, -1), 0.5, colors.grey),
    ("ALIGN", (1, 0), (-1, -1), "RIGHT"),
    ("VALIGN", (0, 0), (-1, -1), "MIDDLE"),
    ("TOPPADDING", (0, 0), (-1, -1), 3),
    ("BOTTOMPADDING", (0, 0), (-1, -1), 3),
]))
story.append(table)
```

Key insight: `generate_report_pdf()` takes `summary: CalculationSummary`. `generate_all_proposals(summary)` is called internally with default constraints. The proposals are generated inside the service, not passed as a parameter.

### Pattern 2: Adding an Excel Sheet via XlsxWriter

The existing `generate_report_xlsx()` calls three `_write_*_sheet()` helpers then closes the workbook. A fourth call is added:

```python
# Source: existing excel_report.py lines 66-70
_write_summary_sheet(wb, summary, header_fmt, bold_fmt, number_fmt, int_fmt)
_write_breakdown_sheet(wb, summary, header_fmt, number_fmt, int_fmt, alt_fmt, alt_right_fmt)
_write_vm_detail_sheet(wb, summary, header_fmt, number_fmt, alt_fmt, alt_right_fmt)
_write_layout_sheet(wb, summary, header_fmt, bold_fmt, number_fmt)  # NEW
wb.close()
```

The `_write_layout_sheet()` function signature must include the workbook so it can create new format objects (XlsxWriter formats are workbook-scoped and cannot be shared across sheets that need new combinations):

```python
def _write_layout_sheet(
    wb: xlsxwriter.Workbook,
    summary: CalculationSummary,
    header_fmt: Format,
    bold_fmt: Format,
    number_fmt: Format,
) -> None:
    from store_predict.pipeline.layout_engine import generate_all_proposals
    proposals = generate_all_proposals(summary)

    ws = wb.add_worksheet(_i18n.t("excel.sheet_layout"))

    # --- Comparison summary at top ---
    # Write strategy name headers in row 0 starting at col 1
    ws.write(0, 0, _i18n.t("layout_page.metric"), header_fmt)
    ws.write(0, 1, _i18n.t("strategy.consolidation"), header_fmt)
    ws.write(0, 2, _i18n.t("strategy.performance"), header_fmt)
    ws.write(0, 3, _i18n.t("strategy.uniform"), header_fmt)

    row = 1
    for metric_key, c_val, p_val, u_val in _layout_metric_rows(proposals):
        ws.write(row, 0, _i18n.t(f"metrics.{metric_key}"), bold_fmt)
        ws.write(row, 1, c_val)
        ws.write(row, 2, p_val)
        ws.write(row, 3, u_val)
        row += 1

    row += 1  # blank separator

    # --- Per-strategy detail sub-tables ---
    for proposal in proposals:
        ws.write(row, 0, _i18n.t(f"strategy.{proposal.strategy_name}"), bold_fmt)
        row += 1
        # DS sub-table headers
        ds_headers = [
            _i18n.t("ds.name"), _i18n.t("ds.raw_cap"), _i18n.t("ds.used"),
            _i18n.t("ds.util"), _i18n.t("ds.vms"), _i18n.t("ds.iops"),
            _i18n.t("ds.workloads"),
        ]
        ws.write_row(row, 0, ds_headers, header_fmt)
        row += 1
        for ds in proposal.datastores:
            ws.write(row, 0, ds.name)
            ws.write(row, 1, ds.raw_capacity_mib / (1024 * 1024), number_fmt)  # TiB
            ws.write(row, 2, ds.used_capacity_mib / (1024 * 1024), number_fmt)
            ws.write(row, 3, ds.utilization_pct, number_fmt)
            ws.write(row, 4, ds.vm_count)
            ws.write(row, 5, ds.total_iops, number_fmt)
            ws.write(row, 6, ", ".join(sorted(ds.workload_types)))
            row += 1
        row += 1  # blank separator between strategies

    ws.freeze_panes(1, 0)
    ws.autofit()
```

### Pattern 3: Shared Metric Row Builder

Both PDF and Excel need the same 15-metric comparison rows. Extract a shared helper that returns tuples of (metric_key, consolidation_val, performance_val, uniform_val) as strings:

```python
def _layout_metric_rows(proposals: list[LayoutProposal]) -> list[tuple[str, str, str, str]]:
    """Build display rows for the strategy comparison table (PDF and Excel shared)."""
    p = proposals  # [consolidation, performance, uniform]
    return [
        ("ds_count",       str(p[0].metrics.total_ds_count), str(p[1].metrics.total_ds_count), str(p[2].metrics.total_ds_count)),
        ("raw_capacity",   _fmt_tib(p[0].metrics.total_raw_capacity_mib), ...),
        ("avg_utilization", _fmt_pct(p[0].metrics.avg_utilization_pct), ...),
        # ... all 15 metrics matching layout_page.py _build_comparison_table()
    ]
```

This helper can live in `pdf_report.py` (and be imported by `excel_report.py`) or in a new thin `services/layout_report.py` module. Given the project convention of keeping services small and focused, placing it in `pdf_report.py` and importing it from `excel_report.py` is acceptable. A dedicated `layout_report.py` is cleaner if the helper grows.

### Anti-Patterns to Avoid

- **Passing proposals as a new API parameter:** Would break existing callers. Generate internally using defaults.
- **Using openpyxl instead of xlsxwriter:** The project uses xlsxwriter for write operations; openpyxl is only for reading input files.
- **Calling `_i18n.set("locale", locale)` twice:** It is already called at the top of `generate_report_pdf()` and `generate_report_xlsx()`; helpers called from within the same function inherit the locale state.
- **Text search in PDF bytes:** CIDFont/FlateDecode encoding means text is NOT searchable in raw PDF bytes. Tests must compare EN vs FR output sizes, not search for string content.
- **Assuming `has_performance_data` for layout:** `generate_all_proposals()` handles both cases internally; do not add a conditional around the layout page.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multi-page PDF | Custom page-break logic | `PageBreak()` flowable in Platypus story list | Platypus handles page breaks, margins, and header callbacks automatically |
| Brand-styled table in PDF | Custom drawing with canvas.rect/drawString | `Table` + `TableStyle` flowable | Platypus handles column widths, padding, alternating rows |
| Excel sheet with multiple sub-tables | Complex offset calculation | Simple `row` counter incremented between sub-tables | XlsxWriter writes to absolute (row, col) ‚Äî just increment a row counter |
| Locale handling in helpers | Passing locale parameter to each helper | `_i18n.t()` / `t()` ‚Äî already set at top of service function | `_i18n.set("locale", locale)` is called once; all subsequent `_i18n.t()` calls use it |
| LayoutProposal metric formatting | Duplicating `_fmt_tib()` / `_fmt_pct()` | Reuse helpers from `layout_page.py` or copy into service | `layout_page.py` has `_fmt_tib()` and `_fmt_pct()` ‚Äî copy (not import, to keep services UI-free) |

## Common Pitfalls

### Pitfall 1: XlsxWriter Format Objects Are Workbook-Scoped

**What goes wrong:** Creating a Format object outside the `wb` workbook (e.g., at module level) causes a runtime error or silent wrong output.
**Why it happens:** XlsxWriter Format objects track their parent workbook internally.
**How to avoid:** Always create formats with `wb.add_format({...})` inside the function that has access to `wb`. Existing helpers in `excel_report.py` receive formats as parameters ‚Äî the same approach must be used for `_write_layout_sheet()`.
**Warning signs:** `AttributeError` or wrong formatting in the produced sheet.

### Pitfall 2: `generate_all_proposals()` Called with Wrong Summary

**What goes wrong:** Calling with an empty `CalculationSummary` returns three empty `LayoutProposal` objects ‚Äî the layout page renders with no data and no errors.
**Why it happens:** The engine handles empty input gracefully (returns `LayoutProposal` with empty `datastores` tuple).
**How to avoid:** Defensive check: if `summary.total_vms == 0`, skip the layout page/sheet rather than rendering an empty table.
**Warning signs:** Layout page appears with only a header row in the comparison table.

### Pitfall 3: Proposals Order Assumption

**What goes wrong:** Accessing `proposals[0]` as consolidation, `proposals[1]` as performance, `proposals[2]` as uniform breaks if the engine order changes.
**Why it happens:** `generate_all_proposals()` currently returns `[consolidation, performance, uniform]` (documented), but order could change.
**How to avoid:** Use `{p.strategy_name: p for p in proposals}` dict lookup, same as `layout_page.py`'s `_build_strategy_tabs()`.
**Warning signs:** Wrong strategy columns in comparison table.

### Pitfall 4: PDF Page 3 Header Duplication

**What goes wrong:** If the header callback draws redundant titles or the `topMargin` is not large enough for the header bar, content overlaps.
**Why it happens:** `_draw_header()` is already wired as `onLaterPages` ‚Äî it runs on every page including the new layout page. The `topMargin = margin + 55` is already accounted for.
**How to avoid:** Do not change `topMargin` or header callbacks. The existing setup is correct for all pages.
**Warning signs:** Content visually overlaps with the header bar on page 3.

### Pitfall 5: Missing i18n Keys

**What goes wrong:** `t("pdf.layout_heading")` or `_i18n.t("excel.sheet_layout")` raises `KeyError` or returns the raw key string.
**Why it happens:** New keys added to `en.yaml` but not `fr.yaml` (or vice versa).
**How to avoid:** Add keys to BOTH locale files in the same commit. Run the existing `test_i18n.py` ‚Äî it verifies key parity between locales.
**Warning signs:** Raw key string appears in PDF/Excel output (e.g., `"pdf.layout_heading"`).

### Pitfall 6: Excel Sheet Count Test Breaks

**What goes wrong:** `test_excel_report.py::TestExcelSheetCount::test_workbook_has_three_sheets` fails because the workbook now has four sheets.
**Why it happens:** The existing test checks for exactly three sheets.
**How to avoid:** Update the test to check for four sheets. Add a new assertion for `sheet4.xml`. The test is in `tests/test_excel_report.py` lines 202-215.
**Warning signs:** CI fails on `test_workbook_has_three_sheets` after implementing the layout sheet.

## Open Questions

1. **Should the layout page appear when `total_vms == 0`?**
   - What we know: `generate_all_proposals()` returns three `LayoutProposal` objects with empty `datastores` tuples when `summary.vm_calculations` is empty.
   - What's unclear: Is a blank layout page acceptable, or should it be skipped?
   - Recommendation: Skip the layout page/sheet if `summary.total_vms == 0` (add `if summary.total_vms == 0: return` guard at start of `_build_layout_page()`). This matches how the UI handles the empty state.

2. **Should `PlacementConstraints` be exposed as a parameter in `generate_report_pdf()`/`generate_report_xlsx()`?**
   - What we know: The report is typically generated from the report page, not the layout page. The layout page stores constraints in `app.storage.tab`.
   - What's unclear: Should the PDF reflect the same constraints the user set on the layout page?
   - Recommendation: Use `PlacementConstraints()` defaults for now. Passing session-stored constraints would require UI changes to pass them to the service layer, which is out of scope for this phase. Note this as a potential future enhancement.

3. **Where should `_layout_metric_rows()` live?**
   - What we know: Both PDF and Excel need identical metric formatting logic.
   - What's unclear: Should it be a module-level function in `pdf_report.py` (imported by `excel_report.py`) or a new `services/layout_report.py`?
   - Recommendation: Since services must not import from UI (`layout_page.py`), duplicate the `_fmt_tib()` and `_fmt_pct()` helpers inside each service (they are trivial one-liners). Put `_layout_metric_rows()` in `pdf_report.py` with an `__all__` export, and import it from `excel_report.py`. This avoids a new file while maintaining separation of UI and services.
