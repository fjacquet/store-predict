# Phase 8.1 Plan 01: LiveOptics ZIP Extraction Summary

**LiveOptics .zip uploaded as fourth format: ZIP extraction + zip bomb guard + 7-test suite, zero changes to existing xlsx/csv paths**

## Performance

## Accomplishments

- New `zip_extraction.py` module with `extract_liveoptics_from_zip(content: bytes) -> tuple[bytes, str]`
- Zip bomb guard: rejects archives whose total uncompressed bytes exceed 100 MB (central directory check, no extraction)
- Pattern matching: `LiveOptics_<id>_VMWARE_<DD>_<MM>_<YYYY>.xlsx` (case-insensitive regex)
- `validation.py` extended to accept "zip" extension and verify PK magic bytes
- `upload.py` patched additively: ZIP branch runs only when filename ends in `.zip`, existing paths unchanged
- Accept prop updated from `.xlsx,.csv` to `.xlsx,.csv,.zip`
- 7-test suite with real in-memory zipfile objects (no mocks) covering all happy/error paths
- 165 tests pass, 1 skipped — zero regressions

## Files Created/Modified

- `src/store_predict/pipeline/validation.py` - Added "zip" to accepted extensions and PK magic bytes check for zip
- `src/store_predict/ui/pages/upload.py` - Added zip import, extraction branch in _handle_upload, updated accept prop
- `tests/test_zip_extraction.py` - 7 tests: happy path, pattern match/reject, no match, invalid zip, multiple members, bomb guard

## Decisions Made

- ZIP extraction runs BEFORE validate_upload so the extracted xlsx bytes go through the existing validation logic unchanged
- Return signature `tuple[bytes, str]` allows the upload handler to use the matched member name for the temp file suffix
- `_XLSX_MAGIC = b"PK\x03\x04"` is already defined in validation.py — ZIP and XLSX share this magic, no new constant needed
- 100 MB uncompressed limit checked via central directory sum (reads headers only, no extraction) — safe zip bomb detection
