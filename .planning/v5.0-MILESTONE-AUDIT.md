## Tech Debt

### Phase 24 — Health Findings Export

1. **Dual PDF implementation (low risk):** `generate_report_pdf()` in `pdf_report.py` has a complete findings implementation (plans 24-01/02) but is not called by any live UI download button. The production PDF path uses `report_print.py` via Playwright. The two implementations are not identical (ReportLab tables vs NiceGUI `ui.table`). They could diverge as findings functionality evolves. Mitigation: `test_pdf_report.py` tests cover `generate_report_pdf()` and will catch regressions; the Playwright path is exercised by the integration flow.

2. **PDF detail table lacks Cluster column (low impact):** `_build_findings_detail()` renders Severity, Category, Finding, Affected VMs — but not Cluster. The cluster data is present in the `HealthFinding` objects (field preserved end-to-end) but not rendered. Users see cluster context on the `/concerns` page and in the Excel Findings sheet. Not a requirement gap (HEXP-02 is silent on cluster column), but the PDF and Excel are asymmetric.

**Total tech debt: 2 items, 0 blockers.**

## Notable Integration Observations

1. **`__no_cluster__` vs `(No Cluster)` sentinel duality** — compute_sizing.py uses `"__no_cluster__"` as an internal groupby key (translated to i18n in the UI). health_checks.py uses `"(No Cluster)"` as a human-readable label for ungrouped-host HW version findings. Both are by design and consistent with the Phase 23 architecture decision documented in 23-01-SUMMARY.md.

2. **`_write_findings_sheet()` called unconditionally** — always invoked in `generate_report_xlsx()`, but internally guards with an early return when `health_result is None / no data / no findings`. Correct pattern; no side-effects from always calling.
