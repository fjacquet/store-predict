### Phase 24 — Health Findings Export

1. **Dual PDF implementation (low risk):** `generate_report_pdf()` in `pdf_report.py` has a complete findings implementation but is not called by any live UI download button. The production PDF path uses `report_print.py` via Playwright. Tests cover both paths. Low divergence risk — any regressions will be caught by `test_pdf_report.py`.

2. **PDF detail table lacks Cluster column (low impact):** `_build_findings_detail()` renders Severity, Category, Finding, Affected VMs — but not Cluster. Cluster data is present in `HealthFinding` objects (preserved through serialization). Users see cluster context on `/concerns` and in the Excel Findings sheet. Not a requirement gap (HEXP-02 is silent on Cluster column).

## Notable Integration Observations

1. **`__no_cluster__` vs `(No Cluster)` sentinel duality** — `compute_sizing.py` uses `"__no_cluster__"` as an internal groupby key (translated to i18n in the UI). `health_checks.py` uses `"(No Cluster)"` as a human-readable label for ungrouped-host HW version findings. Both are by design and consistent with the Phase 23 architecture decision.

2. **`_write_findings_sheet()` called unconditionally** — always invoked in `generate_report_xlsx()`, but internally guards with an early return when `health_result is None / no data / no findings`. Correct pattern; no side-effects.

3. **Per-cluster breakdown table guard** — requires `len(real_clusters) >= 2` (sentinel excluded). Single-cluster files show no breakdown table — correct behavior, documented in Phase 23.

4. **vMSC split ratio clamped [0.01, 0.99]** — `compute_sizing()` clamps input to prevent 0/100 edge cases that would assign all VMs to one site. UI enforces 1–99 range; pipeline has defense-in-depth.
