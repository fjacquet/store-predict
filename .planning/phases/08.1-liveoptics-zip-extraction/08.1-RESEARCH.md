# Phase 08.1: LiveOptics ZIP Extraction - Research

**Researched:** 2026-02-20
**Domain:** Python zipfile stdlib, upload validation, pipeline pre-processing
**Confidence:** HIGH

## Summary

This phase inserts a pure pre-processing step before the existing ingestion pipeline.
When a user uploads a `.zip` file, the system must detect it, extract the LiveOptics
XLSX file matching a known filename pattern, and pass the extracted bytes through the
existing validation and ingestion path unchanged.

The Python `zipfile` stdlib provides everything needed: in-memory extraction from
`BytesIO` without touching disk, listing of members with their uncompressed sizes
before extraction (enabling zip bomb protection), and reliable `BadZipFile` exception
on corrupt input. No third-party library is required.

The integration point is `_handle_upload` in `upload.py`, after `content = await
e.file.read()` and before `validate_upload()`. A single helper function
`extract_liveoptics_from_zip(content: bytes) -> bytes` handles detection (`.zip`
extension OR ZIP magic bytes), size guard, pattern match, and extraction. The existing
`IngestionError` path surfaces errors to the user via `ui.notify()` unchanged.

**Primary recommendation:** Implement a standalone `pipeline/zip_extraction.py` module
with one public function. Call it in `_handle_upload` between file read and validation.
Patch `validate_upload` to accept `.zip` extension with ZIP magic bytes. Add i18n keys
to both locale YAML files.

### Pattern 1: In-Memory ZIP Extraction

**What:** Open a ZIP from raw bytes via `BytesIO`, check total uncompressed size,
match filename pattern, read matched member bytes — all without touching disk.

**When to use:** Upload handler has `bytes`; no temp file yet exists at this point.

**Example:**
```python
# Source: Python 3.x stdlib docs — zipfile module
import io
import re
import zipfile

_LIVEOPTICS_PATTERN = re.compile(
    r"LiveOptics_\d+_VMWARE_\d{2}_\d{2}_\d{4}\.xlsx"
)
_MAX_UNCOMPRESSED_BYTES = 50 * 1024 * 1024  # 50 MiB

def extract_liveoptics_from_zip(content: bytes) -> bytes:
    """Extract LiveOptics XLSX bytes from a ZIP archive in memory.

    Args:
        content: Raw bytes of the .zip file.

    Returns:
        Raw bytes of the matching .xlsx file.

    Raises:
        IngestionError: If ZIP is corrupt, oversized, or contains no matching file.
    """
    try:
        zf_obj = zipfile.ZipFile(io.BytesIO(content))
    except zipfile.BadZipFile as exc:
        raise IngestionError("Uploaded ZIP file is corrupt or not a valid ZIP.") from exc

    with zf_obj as zf:
        # Zip bomb guard: check total uncompressed size before extraction
        total_uncompressed = sum(info.file_size for info in zf.infolist())
        if total_uncompressed > _MAX_UNCOMPRESSED_BYTES:
            raise IngestionError(
                f"ZIP content exceeds the 50 MiB limit "
                f"({total_uncompressed // (1024*1024)} MiB uncompressed)."
            )

        matches = [
            name for name in zf.namelist()
            if _LIVEOPTICS_PATTERN.search(name)
        ]
        if not matches:
            raise IngestionError(
                "No LiveOptics VMWARE file found in ZIP. "
                "Expected a file matching: LiveOptics_<ID>_VMWARE_<DD>_<MM>_<YYYY>.xlsx"
            )

        return zf.read(matches[0])
```

### Pattern 2: Upload Handler Integration

**What:** Intercept `.zip` uploads in `_handle_upload` before routing to
`validate_upload`, extract XLSX bytes, then continue with the existing path.

**When to use:** Only when `e.file.name` ends with `.zip`.

**Example:**
```python
# In _handle_upload, AFTER: content = await e.file.read()
# BEFORE: validate_upload(content, e.file.name)

    content = extract_liveoptics_from_zip(content)
    filename = filename.replace(".zip", ".xlsx")  # or use matched name

validate_upload(content, filename)
# ... rest of existing flow unchanged
```

### Pattern 3: Validation Patch

**What:** Accept `.zip` extension with ZIP magic bytes (`PK\x03\x04`) in
`validate_upload`. ZIP magic bytes are identical to XLSX magic bytes — both are
ZIP-based formats — so the existing `_XLSX_MAGIC` constant reuses directly.

**When to use:** Any `.zip` upload reaches `validate_upload` (defensive check).

**Example:**
```python
# In validation.py — add "zip" branch alongside "xlsx"
_ZIP_MAGIC = b"PK\x03\x04"  # Already defined as _XLSX_MAGIC; reuse or alias

if ext not in ("xlsx", "csv", "zip"):
    raise IngestionError(...)

if ext == "zip" and (len(content) < 4 or content[:4] != _ZIP_MAGIC):
    raise IngestionError("File does not appear to be a valid ZIP file.")
```

### Pattern 4: In-Memory ZIP Test Fixture

**What:** Build a ZIP in memory for tests — no files on disk needed.

**When to use:** All tests for `extract_liveoptics_from_zip` and validation of `.zip`.

**Example:**
```python
# Source: Python stdlib — zipfile.ZipFile with BytesIO
import io
import zipfile

def _make_zip(filename: str, xlsx_bytes: bytes) -> bytes:
    buf = io.BytesIO()
    with zipfile.ZipFile(buf, "w", compression=zipfile.ZIP_DEFLATED) as zf:
        zf.writestr(filename, xlsx_bytes)
    return buf.getvalue()

_VALID_XLSX_MAGIC = b"PK\x03\x04" + b"\x00" * 50

# Usage in tests
zip_bytes = _make_zip("LiveOptics_99999_VMWARE_15_02_2026.xlsx", _VALID_XLSX_MAGIC)
```

### Anti-Patterns to Avoid

- **Extracting to disk first:** No temp file needed; `zf.read(name)` returns `bytes`
  directly. Writing to disk just to read back is unnecessary I/O and cleanup burden.
- **Checking magic bytes only for ZIP detection:** Always also check the extension;
  the upload handler uses the filename to decide whether to call the extractor at all.
- **Raising raw `zipfile.BadZipFile` to the UI:** Always wrap in `IngestionError`
  so the NiceGUI `except IngestionError` handler catches it and shows a friendly notify.
- **Using `re.match` instead of `re.search`:** ZIP members may have directory prefixes
  (e.g., `exports/LiveOptics_...xlsx`). `re.search` handles this; `re.match` does not.
- **Using `re.IGNORECASE` on the pattern:** The real LiveOptics filename format is
  mixed-case exactly as specified. Case-insensitive matching risks accepting malformed
  filenames. Verified pattern works without IGNORECASE flag.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ZIP reading | Custom byte parsing | `zipfile.ZipFile` | Handles compression, CRC, offsets |
| Zip bomb detection | Streaming byte counter | `info.file_size` pre-read | Already in ZIP central directory; no extraction needed |
| Filename pattern | Custom string parsing | `re.compile` + `re.search` | Handles subdirectory paths in zip |

**Key insight:** The ZIP central directory already stores uncompressed sizes for all
members. Reading `info.file_size` before calling `zf.read()` is O(n_members), not O(bytes),
making zip bomb detection essentially free.

## Common Pitfalls

### Pitfall 1: ZIP magic bytes identical to XLSX magic bytes
**What goes wrong:** Developer assumes `.zip` needs a different magic check than `.xlsx`
and adds new constant, creating confusion or duplication.
**Why it happens:** Both `.xlsx` and `.zip` are ZIP-based formats; their first 4 bytes
are identical: `PK\x03\x04`.
**How to avoid:** Reuse `_XLSX_MAGIC` (or rename to `_ZIP_MAGIC`) for both `.zip` and
`.xlsx` validation. Add a comment explaining why.
**Warning signs:** Two constants with value `b"PK\x03\x04"` in the file.

### Pitfall 2: NiceGUI upload widget `accept` prop not updated
**What goes wrong:** Backend accepts `.zip` but frontend widget filters it out, so users
cannot select `.zip` files in the file picker.
**Why it happens:** The widget has `.props('accept=".xlsx,.csv"')` hardcoded.
**How to avoid:** Update `upload.py` line 100 to `.props('accept=".xlsx,.csv,.zip"')`.
**Warning signs:** Backend tests pass but manual upload of `.zip` is blocked by browser.

### Pitfall 3: `filename` variable not updated after extraction
**What goes wrong:** After extracting XLSX bytes from ZIP, `validate_upload` is called
with the original `.zip` filename, failing the extension check.
**Why it happens:** Extraction replaces `content` bytes but not `filename`.
**How to avoid:** After extraction, set `filename` to the matched member name (which
ends in `.xlsx`) or to `original_name.replace(".zip", ".xlsx")`.
**Warning signs:** `validate_upload` raises "Unsupported file type: .zip" even after
adding ZIP support, because the extracted content is XLSX but filename is still `.zip`.

### Pitfall 4: Test zip fixture uses wrong compression
**What goes wrong:** `zipfile.ZipFile(buf, "w")` defaults to `ZIP_STORED` (no
compression), producing different bytes than real LiveOptics ZIPs.
**Why it happens:** Tests may pass regardless (compression type transparent to reader),
but the fixture is brittle if tested against a compression-sniffing validator.
**How to avoid:** Use `compression=zipfile.ZIP_DEFLATED` in test fixtures to be
realistic.

### Pitfall 5: Omitting i18n keys causes KeyError at runtime
**What goes wrong:** Error messages from `extract_liveoptics_from_zip` are hardcoded
English strings, not wrapped in `t()`.
**Why it happens:** The extractor lives in `pipeline/`, not `ui/`, and `t()` should
not be called from pipeline code (no NiceGUI context).
**How to avoid:** Pipeline layer raises `IngestionError` with hardcoded English strings
(as all existing pipeline errors do). The UI layer (`upload.py`) catches `IngestionError`
and calls `ui.notify(str(exc))` — which is already i18n-neutral since `IngestionError`
messages are not translated, consistent with existing code. Add i18n keys only to
things the UI layer constructs directly.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Extract to temp dir | Extract to `BytesIO` in memory | Python 3.x (always available) | No cleanup, no disk path |

**Deprecated/outdated:**
- `zipfile.ZipFile` with `allowZip64=False` default: old Python versions. Python 3.4+
  defaults `allowZip64=True`, which handles large ZIPs. No action needed.

## Open Questions

1. **Multiple matching files in one ZIP**
   - What we know: pattern says "always this exact format" — one file expected
   - What's unclear: What if a ZIP has two LiveOptics files (e.g., incremental exports)?
   - Recommendation: Use the first match (`matches[0]`). Document this choice. The
     phase description says the pattern is "always exactly this format."

2. **i18n for IngestionError messages in pipeline layer**
   - What we know: All existing `IngestionError` messages in pipeline code are hardcoded
     English strings. The UI catches them and passes `str(exc)` directly to `ui.notify`.
   - What's unclear: Should new messages follow this pattern or be keyed strings?
   - Recommendation: Follow existing convention — hardcoded English strings in pipeline
     layer. No `t()` calls from `pipeline/` modules.
